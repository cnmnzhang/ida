## 3. Diagnosis-Based Anemia

### What SNOMED codes Synthea generated

Only one anemia-related SNOMED code appears in this cohort:

| SNOMED | Description | Patients |
|---|---|---|
| 271737000 | Anemia (disorder) | 390 (34.7%) |

### Iron Deficiency Anemia codes: absent

The IDA-specific codes — **87522002** (Iron deficiency anemia) and **234347009** (Anemia due to iron deficiency) — were **not generated**. This is the expected behavior of Synthea's default anemia module:

- The primary anemia module fires on a population-prevalence curve and codes to the generic `271737000`.
- The `iron_deficiency_anemia` sub-module, which would emit the specific codes, has a low base firing probability and did not fire for any patient in this seed.
- Increasing population size, adjusting the seed, or explicitly invoking the sub-module via `-m iron_deficiency_anemia` are the standard approaches to obtain IDA-specific codes.

## 5. Iron Panel and Ferritin

### Correction to prior documentation

The `src/synth/README_synthea.md` previously stated: *"Synthea does not generate ferritin"*. This is **incorrect** for this cohort.

When Synthea's anemia module fires and orders iron studies, it generates a full iron panel:

| LOINC | Description | Observations |
|---|---|---|
| 2276-4 | Ferritin (Serum/Plasma) | 473 |
| 2498-4 | Iron (Serum/Plasma) | 337 |
| 2500-7 | Iron binding capacity | 337 |
| 2502-3 | Iron saturation | 337 |

### Ferritin distribution

| Stat | Value |
|---|---|
| Mean | 203.0 µg/L |
| Std dev | 189.0 µg/L |
| Median | 114.6 µg/L |
| Min | 30.1 µg/L |
| Max | 1,128.4 µg/L |
| P(ferritin < 30 µg/L) | 0.0% |
| P(ferritin < 12 µg/L) | 0.0% |

The minimum ferritin value is exactly 30.1 µg/L — Synthea floors ferritin at the lower bound of the normal range. **No patient has a below-threshold ferritin value** (clinical iron deficiency is typically defined as ferritin < 30 µg/L). This means ferritin, while present, carries no discriminative IDA signal in this synthetic dataset.

## 6. Diagnostic Gap Analysis

The diagnostic gap compares lab-defined anemia (WHO criterion applied to most-recent HB) against the presence of any anemia diagnosis code in `conditions.csv`.

```
                     Has anemia Dx?
                     Yes      No
Lab anemia  Yes  │   75   │   43   │  118
            No   │  315   │  692   │  1007
                 │  390   │  735   │  1125
```

| Segment | Count | Share of cohort |
|---|---|---|
| Lab anemia + diagnosis (concordant) | 75 | 6.7% |
| Lab anemia, no diagnosis ("underdiagnosed") | 43 | 3.8% |
| Diagnosis, HB now normal ("resolved") | 315 | 28.0% |
| Neither | 692 | 61.5% |

### The underdiagnosis signal (43 patients)

These patients have a most-recent HB below the WHO threshold but no anemia code on record. This is the synthetic analog of clinical underdiagnosis. The gap rate of **3.8%** should be interpreted cautiously:

- Synthea's diagnosis coding is driven by module firing rules, not a model of clinical documentation practices. A real EHR would reflect ordering patterns, visit frequency, and physician coding behavior.
- Some gap patients may have had their anemia develop only recently in the simulation, before a follow-up encounter would have generated the diagnosis.

### The "resolved" group (315 patients)

315 patients carry an anemia diagnosis but have HB above the WHO threshold at their most recent measurement. This reflects Synthea's longitudinal fidelity — anemia conditions persist in `conditions.csv` after the episode, matching how diagnosis codes accumulate in real problem lists. This is the larger and analytically more interesting group for understanding the mismatch between active diagnoses and current lab status.

## 7. Synthea Modeling Limitations Relevant to IDA Research

| Limitation | Impact on this analysis |
|---|---|
| No IDA-specific SNOMED codes generated | Cannot distinguish IDA from generic anemia via diagnosis codes in this cohort |
| Ferritin floored at normal lower bound (~30 µg/L) | Ferritin cannot be used to identify iron deficiency; it has no sub-threshold values |
| Iron panel only ordered during anemia module encounters | Iron studies are tied to the anemia workflow; incidental iron panels do not appear |
| Diagnosis codes persist after resolution | Most-recent HB and active diagnoses are misaligned in 28% of the cohort |

## 8. Next Steps

1. **Obtain IDA-specific codes** — run with the `iron_deficiency_anemia` sub-module explicitly enabled (`-m iron_deficiency_anemia`) or use a larger population (`-p 5000+`) to increase the probability of IDA module firing.

2. **Ferritin is not useful as-is** — since Synthea floors ferritin, a simulated ferritin (as implemented in `curate_data.py`) calibrated to HB drop from setpoint is more appropriate for IDA discrimination tasks.

3. **Revisit the diagnostic gap using encounter timing** — restrict the gap analysis to patients who had a clinical encounter after the most-recent low HB reading, to filter out patients whose low HB is simply not yet coded.

4. **Stratify by sex correctly** — the male-heavy lab anemia finding is a threshold artifact. Future analyses should report sex-stratified prevalence and consider whether the downstream ML models are sex-stratified or use sex as a feature.
